테스트 데이터 작성 규칙
===

Cucumber Feature 파일 작성 시 일관성을 유지하기 위한 테스트 데이터 규칙입니다.

1. 사용자 정보
---

### 이메일
- **기본 사용자**: `user@email.com`
- **여러 사용자 필요 시**: `userA@email.com`, `userB@email.com`, `userC@email.com`, ...

### 비밀번호
- **모든 사용자 공통**: `password`

2. 경기 및 장소
---

### 경기장
- **기본**: `잠실 야구장` (정식 명칭: 잠실 종합운동장 야구장)

### 팀
- **기본 홈팀**: `LG 트윈스`
- **기본 원정팀**: `두산 베어스`

3. 좌석 및 점유 (Allocation)
---

### 영역 (Area)
- **내야**: 1루/3루 응원석 쪽
- **외야**: 외야석 쪽

### 점유 용어
- **점유 (Hold)**: 좌석을 임시 선점함. (Gherkin: "점유", "점유를 요청하면")
- **반납 (Release)**: 점유한 좌석을 다시 공석으로 돌림. (Gherkin: "반납", "반납 요청하면")

### 구역 이름 (Block Name)
- **형식**: `{영역}-{진영}-{번호}`
- **예시**: `내야-연고-1`, `외야-좌측-1`

### 좌석 위치 (Seat Position)
- **형식**: `{n}행 {m}열`
- **예시**: `1행 1열`, `10행 5열`

4. 시나리오 ID (matchId)
---
- **별칭(Alias)**: 시나리오 본문에서 경기를 특정할 때 `0`, `1`, `2` 등의 숫자를 사용합니다.
- **예시**: `경기 0`, `주어진 경기`

5. 일관성 체크리스트
---

feature 파일 작성 시 다음을 확인하세요:

- [ ] "점유", "반납" 용어를 정확히 사용했는가?
- [ ] 구역 이름이 `내야-연고-1`과 같은 형식을 따르는가?
- [ ] 좌석 위치가 `n행 m열` 형식을 따르는가?
- [ ] "~석" 접미사가 제거되었는가? (필요한 경우 제외)
