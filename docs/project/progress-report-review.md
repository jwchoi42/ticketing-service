## A. 보고서 정확성 문제 (팩트 체크)

### 1. 부하 테스트 성능 목표 수치 불일치

**Section 6**에 기재된 성능 목표:
> p50 < 100ms, p95 < 300ms, p99 < 500ms, TPS > 500

**실제 K6 Threshold** (`20260201-171759-summary.md`):
> p95 < 1000ms, p99 < 3000ms, error rate < 1%

두 수치가 다르다. "이상적 목표"와 "테스트 임계값"이 다른 것이라면 구분을 명시해야 한다.

### 2. 부하 테스트 결과 누락

보고서에 "K6 스크립트 작성 및 실행 완료"라고만 적혀 있고, **실제 측정 결과가 없다.** 4가지 전략 비교 테스트까지 했는데 결과가 빠져 있다.

최소한 포함해야 하는 것:
- 채택 전략: `denormalized + collapsing`
- 결과: TPS 573, avg 292ms, p95 1,095ms, error 0.008%
- 목표(TPS > 500) 달성 여부

### 3. 현재 채택 전략 미명시

보고서 어디에도 "denormalized + collapsing을 채택했다"는 내용이 없다. 이 보고서를 AI 에이전트에 전달하면, 에이전트는 현재 어떤 전략이 운영 중인지 알 수 없다.

## C. 기획 관점 지적 사항 (포트폴리오 목적 기준)

### 6. MVP 범위 미정의 — 가장 큰 문제

현재 보고서는 "만든 것"의 목록이지, "완성이란 무엇인가"의 정의가 아니다. `feature-roadmap.md`가 별도로 있지만 progress report와 연결되지 않아, **현재 개발이 어디를 향하고 있는지** 알 수 없다.

포트폴리오용이라면 특히 중요한 것:
- 이 프로젝트에서 **보여주고 싶은 핵심 역량**이 무엇인지
- 그 역량을 증명하기 위해 **반드시 완성해야 하는 기능**이 무엇인지
- 명시적으로 **하지 않기로 한 것**(Out-of-scope)은 무엇인지

### 7. 사용자 여정(User Journey)이 끊겨 있음 — 포트폴리오 치명적

```
회원가입 → 로그인 → 경기 선택 → 좌석 선점 → 예약 → 결제(Mock) → ???
```

**끊기는 지점:**
- 결제 후 사용자가 보는 화면이 없음 (티켓 페이지가 껍데기)
- 결제 전용 UI가 없어서 "결제하고 있다"는 인식 없이 API만 호출됨
- 예약 취소 불가 → 실수로 예약해도 탈출구 없음

포트폴리오 데모에서 "결제 확인까지 했는데 그 다음 화면이 없다"면, 기술적으로 아무리 잘 만들어도 **미완성 인상**을 준다. E2E 흐름의 끝을 보여주는 것이 중요하다.

### 8. 인증 미구현 — 포트폴리오 관점 재평가 필요

`userId` 파라미터 전달 방식은 실서비스에서는 치명적이지만, **포트폴리오 목적이라면 다르게 볼 수 있다:**

- 인증/인가는 "이 프로젝트의 핵심 역량"인가? (동시성 제어, 실시간 SSE, 성능 최적화가 핵심이라면 인증은 부차적)
- 다만 면접에서 "인증은 왜 안 했나?"는 **반드시 나올 질문**이므로, 의도적 제외라면 보고서에 **왜 제외했는지 근거**를 남겨야 함
- JWT 구현 자체는 난이도가 높지 않으므로, 완성도 측면에서 구현하는 것이 유리

### 9. 가격 체계 부재

170,000원 하드코딩은:
- 도메인 모델에 가격 개념이 아예 없다는 의미
- Payment의 amount 출처가 불명확
- 결제 금액 정합성 검증 불가

포트폴리오 관점: 최소한 Block 단위 가격 데이터를 시딩하고, 결제 시 서버에서 금액을 계산하는 흐름이 있어야 "결제 도메인을 이해하고 있다"는 인상을 줄 수 있다.

### 10. 만료 좌석 회수 — 프로젝트 핵심 목표와 모순

프로젝트 핵심: **"좌석 선점 실패 경험 최소화"**

현재 "반응적 회수"(다른 사용자가 해당 좌석 선점 시도 시에만 만료 체크)의 문제:
- 인기 없는 좌석은 영원히 HOLD로 보임
- SSE로 HOLD 상태가 브로드캐스트되므로 다른 사용자가 선택 자체를 안 함
- **프로젝트 핵심 문제의식과 정면으로 모순**

면접에서 "선점 실패를 줄인다고 했는데, 만료된 좌석이 계속 HOLD로 보이면 오히려 사용 가능한 좌석이 줄어드는 것 아닌가?"라는 질문에 답이 필요하다.
